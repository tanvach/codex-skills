# Build Cross-Agent Skill Sync and Installation Workflow

## Goal

Make this repository the single source of truth for personal agent skills so one repo can drive both Codex and Claude Code. After the change, a user should be able to update a Codex skill in this repo, run one sync command from either tool, and have Codex skills linked into `~/.codex/skills` and Claude slash commands generated and linked into `~/.claude/commands` with predictable install, update, and prune behavior.

## Scope

In scope are: restructuring the repo around explicit scripts, generating Claude-compatible command files from Codex `SKILL.md` sources, installing and pruning both Codex and Claude outputs, and adding a `sync-skills` entry point that can orchestrate the scripts from an agent window. Also in scope are repo documentation updates that explain the new workflow and generated-artifact model. Out of scope are support for every possible Codex skill feature in Claude, deep semantic rewriting of complex multi-file skills, and any attempt to make Claude consume Codex `SKILL.md` directories directly without generation. This plan assumes Codex skill folders remain the source of truth and Claude commands are generated artifacts derived from them.

## Current State

The repo currently contains only top-level Codex skill folders such as `/Users/tanvach/GitHub/codex-skills/double-check-work`, `/Users/tanvach/GitHub/codex-skills/execplan-create`, `/Users/tanvach/GitHub/codex-skills/implement-execplan`, `/Users/tanvach/GitHub/codex-skills/architecture-docs-creator`, and `/Users/tanvach/GitHub/codex-skills/update-architecture-docs`. Codex installation is handled by a single root script at `/Users/tanvach/GitHub/codex-skills/install.sh`, which discovers top-level folders containing `SKILL.md`, links them into `~/.codex/skills`, and optionally prunes stale repo-managed symlinks. The repo README at `/Users/tanvach/GitHub/codex-skills/README.md` documents only the Codex install flow. There is no `scripts/` directory, no generated output directory for Claude, no Claude command install flow, and no `sync-skills` orchestrator skill or Claude command.

## Target State

The repo should expose three deterministic scripts and one orchestration layer. Codex skill folders at the repo root remain the source of truth. A generator script converts those sources into Claude slash-command Markdown files under a generated directory such as `/Users/tanvach/GitHub/codex-skills/generated/claude-commands/`. A Codex installer script links and prunes source skill folders into `~/.codex/skills`. A Claude installer script links and prunes generated command files into `~/.claude/commands` without touching unrelated hand-authored Claude commands. A `sync-skills` Codex skill and a matching generated Claude `/sync-skills` command should run the scripts in the right order, offer a `--check` or dry-run path, and summarize what changed or what needs manual attention. The generated Claude commands should strip Codex YAML frontmatter, avoid Codex-only metadata such as `agents/openai.yaml`, and rewrite obvious Codex-specific inline references like `$double-check-work` into Claude-friendly wording.

## Implementation Plan

First, reorganize the repository around explicit script and generated-artifact boundaries. Add `/Users/tanvach/GitHub/codex-skills/scripts/` and move the current root installer logic from `/Users/tanvach/GitHub/codex-skills/install.sh` into `/Users/tanvach/GitHub/codex-skills/scripts/install_codex_skills.sh`. Keep the current behaviors that already work well: discover top-level directories that contain `SKILL.md`, back up conflicting local destinations before replacing them, and support prune mode that removes stale symlinks only when they point back into this repo. Leave a thin root-level `/Users/tanvach/GitHub/codex-skills/install.sh` wrapper in place for backward compatibility; it should call the new Codex installer so existing muscle memory and README examples do not break.

Next, add a deterministic Claude generator at `/Users/tanvach/GitHub/codex-skills/scripts/create_claude_skills.sh`. This script should enumerate the same top-level source skill folders, read each `SKILL.md`, strip YAML frontmatter, and write one generated Markdown command per skill into `/Users/tanvach/GitHub/codex-skills/generated/claude-commands/<skill-name>.md`. Preserve the body content as much as possible so Codex remains the source of truth, but normalize a few known differences in a predictable way: remove Codex metadata blocks, ignore `agents/openai.yaml`, and rewrite explicit `$skill-name` mentions into Claude-friendly text such as `/skill-name` or “this command,” depending on context. If a source skill later needs Claude-specific wording, reserve an override convention such as `<skill-dir>/claude.override.md`; the generator should prefer that file when present and otherwise fall back to transformed `SKILL.md`.

Then, add `/Users/tanvach/GitHub/codex-skills/scripts/install_claude_skills.sh` to link generated Claude command files into `~/.claude/commands`. This script should mirror the Codex installer’s backup and prune behavior but operate on Markdown files instead of directories. It should never delete or move unrelated Claude commands; prune should only remove symlinks whose targets point into `/Users/tanvach/GitHub/codex-skills/generated/claude-commands/`. The installer should create `~/.claude/commands` if it does not exist and print clear counts for linked, skipped, backed up, and pruned commands. Add a dry-run or `--check` mode here if it can be implemented consistently across both installers and the generator.

After the script layer is in place, add orchestration so the whole workflow can be run from either agent window. Create a new Codex skill at `/Users/tanvach/GitHub/codex-skills/sync-skills/SKILL.md` that tells Codex to inspect the repo, run the generation and install scripts in order, and summarize the results rather than editing generated files by hand. The skill should support normal mode and a non-mutating check mode. Because Claude cannot consume Codex skills directly, ensure the generator also emits `/Users/tanvach/GitHub/codex-skills/generated/claude-commands/sync-skills.md`, or provide a checked-in override source so Claude gets a concise slash command that instructs Claude to run the same scripts. The orchestrator should remain a thin wrapper over scripts; file mutations must stay inside scripts so behavior stays deterministic and debuggable.

Finally, update repository documentation and examples. Expand `/Users/tanvach/GitHub/codex-skills/README.md` to explain the source-of-truth model, the generated Claude output directory, the purpose of each script, the difference between install and prune, and how to sync on a new machine. Include concrete flows for “Codex only,” “Claude only,” and “sync both.” Document what happens when a new skill is added: commit the new source folder, rerun the generator and installers, restart the relevant tool, and optionally run prune after deletions. Also document the override convention and the rule that generated Claude files should not be edited directly.

## Risks

The biggest risk is silent semantic drift between Codex skill instructions and Claude command behavior. Mitigate this by keeping the transformation rules intentionally small, deterministic, and documented, and by supporting explicit `claude.override.md` files for the minority of skills that need manual tuning. A second risk is overly aggressive prune logic deleting user-managed commands or skills. Mitigate that by pruning only symlinks whose targets resolve inside this repo’s managed source or generated directories. A third risk is that the orchestration skill starts mutating generated files directly instead of relying on scripts, which would make the system harder to reason about. Mitigate that by keeping the skill instructions explicit that scripts own all generated output and installs. A final risk is future Codex or Claude format changes. Mitigate that by keeping generation logic centralized in one script, validating generated output with spot checks, and documenting the override path so platform-specific changes do not force a full redesign.
